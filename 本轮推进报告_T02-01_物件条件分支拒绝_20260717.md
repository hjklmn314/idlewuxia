# 本轮推进报告：T02-01 物件条件分支拒绝

## 结果

T02-01 已完成代码施工与本地自动验收。物件动作存在精确配置分支、但所有分支条件均不满足时，运行时现在返回
`interactableInteractionRejected`，不再回退执行 `exactBranches[0]`。

聚焦门禁 `npm run runtime:condition-negative` 为 2/2 通过：

- false 路径：`玩家物品大于` 条件不满足，动作明确拒绝，玩家、背包、运行时 flags、地图标记和当前物件选择均不变化；
- true 路径：同一配置仅改变输入物品数量后正常执行，并按配置发放结果物品。

机器报告位于忽略目录
`outputs/t02_01_interactable_condition/interactable_branch_tests.json`，不进入 Git。

## 根因

`branchForInteractableAction` 先寻找条件通过的精确分支，但查找失败后继续使用
`exactBranches[0]`。调用方又没有 NPC 交互已有的 availability 拒绝层，因此失败条件分支仍会进入
`applyResultEffects`，最终以 `accepted=true` 返回并产生配置副作用。

## 模块化与数据驱动

本轮新增内部深模块 `configuredBranchAvailability`，其小接口统一封装：

- 候选配置分支遍历；
- `conditionTokens -> conditionLookup -> evaluateConditionToken` 求值；
- `available / reason / checks / conditionTokens / evidenceLevel` 结果模型；
- 未配置条件分支与已配置但不满足分支的区分。

NPC 和物件 availability 均复用该模块。具体物件 ID、条件阈值、奖励 ID 和结果数值仍只存在于 JSON/测试夹具，
运行时代码只负责通用解释、拒绝、状态路由和结果执行。快照新增
`selectedInteractableActionAvailability`，为 UI 后续禁用按钮和展示配置要求提供稳定数据接口。

完整执行链为：

`interactable.actions -> exact action branches -> configuredBranchAvailability -> branchConditionsMet
-> evaluateConditionToken -> rejected 或 branchForInteractableAction -> applyResultEffects`。
