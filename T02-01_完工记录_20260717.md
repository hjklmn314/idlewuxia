## 2. 二次审计发现的问题

1. 多个分支同时满足时，availability 按配置顺序取第一个分支，执行器却优先取有 narrative 的分支，导致条件、证据和副作用可能来自不同分支。
2. Runtime 已输出 `selectedInteractableActionAvailability`，但 Web 物件面板不消费它；不可用动作仍显示为普通可点击按钮。
3. `interactableInteractionRejected` 未进入房间日志，且 UI 入口在 rejected 时不重新渲染，玩家看到静默失败。
4. 原测试使用错误的 `id/screen` 字段、只有单分支单次用例、报告计数硬编码为 3，不能证明重复执行、存档重载和多分支一致性。
5. 原完工报告缺少强制要求的行号、退出码、人工验收、风险、未覆盖项与回滚方式。

## 3. 修改方案

- 将条件评估与分支选择收口到 `configuredBranchDecision()`，一次返回 `branch + available + checks + evidenceLevel`。
- `interactableActionDecision()` 成为物件动作的单一决策 seam；snapshot 只投影 availability，执行器直接消费同一 decision 的 branch。
- 保持配置顺序优先，不允许 narrative 是否存在改变分支优先级。
- Web 面板基于 snapshot 呈现锁定样式、可见条件、`aria-disabled` 与原生 `disabled`。
- rejected 事件进入房间日志；UI 调用后无论 accepted/rejected 都重新渲染。
- 回归测试改为行为列表驱动，报告数量由实际 case 数计算。

## 6. 代码变化

完整链路：

`interactable.actions -> interactableBranchesForAction -> configuredBranchDecision -> branchConditionsMet -> selected branch / rejected -> UI availability projection or applyResultEffects`

模块接口约束：

- 同一组输入必须产生唯一分支 decision。
- availability 与 execution 不得各自实现分支优先级。
- rejected 只允许追加审计事件，其余运行时状态不变。
- UI 不推断条件，只消费 Runtime snapshot。
