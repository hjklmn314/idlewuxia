# 本轮推进报告：开发前置分析与配置驱动门禁

日期：2026-07-10

## 结论

后续开发不能再从 UI 或临时逻辑开始。每个新内容必须先读取当前项目代码、竞品 Lua/配置、已有分析报告和验收产物，形成证据表与配置契约，再进入设计、资产制作、代码开发、验收和审计。

核心原则：

- 竞品录像只做界面布局和交互节奏参考，真实流程以竞品代码和配置为准。
- 章节、地图、房间、NPC、敌人、剧情、奖励、门槛、资产引用、UI 文案、数值参数必须先进入 JSON/CSV/DataTable。
- 代码只负责解释配置、路由动作、维护状态机、渲染视图、执行通用算法和输出审计。
- 未解析的机制不能猜成已实现；必须标为 Lua 证据、配置证据、反推测试、项目设计、未知。

## 开发前必须完成的分析顺序

1. 逐行阅读当前活跃入口代码，确认哪些是框架，哪些是临时兼容。
2. 读取对应竞品 Lua/配置，记录来源文件、字段和值。
3. 将结论写入 evidence ledger，区分 `lua_confirmed`、`config_confirmed`、`cross_source_confirmed`、`reverse_tested`、`design_proposal`、`unknown`。
4. 先补配置表，再扩展通用解释器；不能在代码里写死章节、NPC、房间、奖励、技能、Buff、资产路径。
5. 资产制作先过资产规范：竞品资源仅限研究占位，产品级资源必须有自有来源、风格状态和绑定行。
6. 验收必须同时通过自动门禁和真实浏览器/PIE/人工证据，不能用离线截图替代真实运行。

## 设计思想

当前项目应该按“深模块 + 小接口”推进：

- `src/wuxiaFirstSessionFlow.js` 是状态机与竞品配置解释器，不应承载具体章节文案。
- `src/wuxia-main.js` 是移动端视图与动作路由，不应承载业务流程真相。
- `config/wuxia_first_session_flow.json` 是当前第一会话与第一章内容事实源。
- `config/wuxia_first_session_screen_contract.json` 是界面结构与动作绑定事实源。
- `outputs/*` 是审计与验收事实，不是手写设计稿。
